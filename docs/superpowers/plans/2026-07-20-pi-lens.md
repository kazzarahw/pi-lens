# pi-lens — Build Spec

> Design/implementation spec. Decisions, responsibilities, interfaces, build
> order, acceptance criteria. No function bodies. This is the largest extension;
> the spec is deep where the real design is (the LSP client) and lean elsewhere.

**Goal:** Real-time code feedback — LSP diagnostics + linter/formatter output injected automatically after the agent reads/writes/edits, plus one `lens` tool (an `action` enum: hover/references/definition/rename) for manual LSP queries, plus a dynamic test/build **verify** pass that runs automatically (absorbed pi-verify — hook-only, no tool).

**Architecture:** An LSP client (JSON-RPC over stdio) and a server pool keyed by language; a linter runner; a pure diagnostics layer that formats `<pi-lens>` injections; a verify runner. Static feedback fires on `tool_result`; the verify pass fires on `agent_settled`. Coordination via `pi.events`.

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, `node:child_process` for LSP servers/linters/tests, `@earendil-works/pi-*` peers. No LSP library — hand-rolled minimal client (framing + the handful of methods needed).

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Application:
- **One tool** `lens` with an `action` `StringEnum` (`hover` | `references` | `definition` | `rename`) — the action-enum-over-N-tools rule (house-style §3). No `lens_verify` tool: verify runs automatically as a hook.
- Injects `<pi-lens>` blocks. Emits `lens:clean { file }` / `lens:issues { file, diagnostics }` and `verify:passed` / `verify:failed { cmd, failures }`.
- Enforcement dial applies. Command `/pi-lens`. Log `[pi-lens]`.

## Design Decisions

1. **Hand-rolled minimal LSP client.** Speak LSP over a server's stdio: `Content-Length` framing, `initialize`/`initialized`, `textDocument/didOpen`/`didChange`, receive `textDocument/publishDiagnostics`, and request `hover`/`rename`/`references`/`definition`. No full LSP lib — only what these features need. The **message framing** is a pure, testable unit.
2. **Server pool keyed by language**, lazily spawned. `config` maps file extension → server command. **Ship TS/JS first** (`typescript-language-server --stdio`); Python (`pyright`/`pylsp`) and others added purely by config. Unknown extension → no LSP (linters may still run).
3. **Diagnostics = LSP ⊕ linters.** After a read/write/edit `tool_result` touching a file, gather that file's current LSP diagnostics plus configured linter output, merge, and inject one `<pi-lens>` block (or emit `lens:clean` if empty). Linters are `{ cmd, parse }` specs per language (eslint, ruff, …).
4. **Verify subsystem (absorbed pi-verify) — hook-only, no tool.** A configured test/build command runs automatically on `agent_settled` (in `notify`/`block`). Parse pass/fail + failing-test names; inject a `<pi-lens>` verify block; emit `verify:passed`/`verify:failed`. **No `lens_verify` tool** — the agent runs tests manually via Bash; verify's whole value is the automatic timing, the emitted event (pi-memory captures a gotcha on `verify:failed`), and the `block`-mode turn gate — all harness concerns. **Coordination:** auto-verify waits for a `lens:clean` (don't run tests while the file won't even parse). `verify.ts` stays a cleanly separable module in case it ever splits back out.
5. **Enforcement dial:**
   - `off` — manual tools only; no injection, no auto-verify.
   - `notify` *(default)* — inject diagnostics on edit + auto-verify on settle; never blocks.
   - `block` — additionally `{ block: true }` a `write`/`edit` `tool_call` whose *previous* result left unresolved **errors** (not warnings), and/or block turn-end while tests are red. Hard mode.
6. **The `lens` tool proxies the LSP server** at a position (`path`, `line`, `col`) — one tool, `action` selecting hover/references/definition/rename (rename adds `new_name`), returning hover text / locations / rename edits.

### Open items for the human
- Default `mode` (recommend `notify`). The verify command per project (autodetect `package.json` test script / `pytest`, else configured). Which languages ship in v1 beyond TS/JS.

## Interfaces / Contracts

```typescript
interface Diagnostic {
  file: string; line: number; col: number;
  severity: "error" | "warning" | "info";
  message: string; source: string; code?: string;
}
interface Position { line: number; col: number; }
interface Location { file: string; line: number; col: number; }

// lsp/framing.ts (pure)
function encodeMessage(obj: unknown): string;                 // Content-Length header + JSON
function decodeMessages(buffer: string): { messages: unknown[]; rest: string };

// lsp/client.ts — stdio injectable ({write, onData}) for tests
interface LspClient {
  initialize(rootUri: string): Promise<void>;
  didOpen(uri: string, text: string, languageId: string): void;
  didChange(uri: string, text: string): void;
  onDiagnostics(cb: (uri: string, ds: Diagnostic[]) => void): void;
  hover(uri: string, pos: Position): Promise<string | null>;
  rename(uri: string, pos: Position, newName: string): Promise<Array<{ file: string; edits: unknown[] }>>;
  references(uri: string, pos: Position): Promise<Location[]>;
  definition(uri: string, pos: Position): Promise<Location[]>;
  shutdown(): Promise<void>;
}
function createLspClient(io: { write: (s: string) => void; onData: (cb: (s: string) => void) => void }): LspClient;

// lsp/manager.ts
interface LspManager {
  forFile(path: string): LspClient | null;      // spawns/reuses by language
  diagnosticsFor(path: string): Diagnostic[];
  shutdownAll(): Promise<void>;
}

// linters.ts
interface LinterSpec { cmd: (file: string) => string[]; parse: (stdout: string, stderr: string) => Diagnostic[]; }
function runLinters(path: string, specs: LinterSpec[], exec: ExecFn): Promise<Diagnostic[]>;

// diagnostics.ts (pure)
function mergeDiagnostics(...groups: Diagnostic[][]): Diagnostic[];
function formatDiagnostics(path: string, ds: Diagnostic[]): string;   // "" when clean

// verify.ts
interface VerifyResult { passed: boolean; failures: string[]; raw: string; }
function parseVerify(stdout: string, stderr: string, code: number): VerifyResult;
function formatVerify(r: VerifyResult): string;                       // <pi-lens> verify block

// tool: lens({ action: "hover"|"references"|"definition"|"rename", path, line, col, new_name? })
//   action is a StringEnum; new_name required only for "rename". Returns hover text / Location[] / rename edits.
// verify: NO tool — runs on agent_settled (hook). events below.
// events: lens:clean {file} | lens:issues {file, diagnostics}; verify:passed | verify:failed {cmd, failures}
```

## Build Sequence

- [ ] **1 · Scaffold** — per house-style. Accept: `bun test` runs.
- [ ] **2 · LSP framing** — Accept: `encodeMessage` emits correct `Content-Length`; `decodeMessages` parses one/multiple/partial buffers and returns leftover `rest`. (Pure; the client's foundation.)
- [ ] **3 · LSP client (fake stdio)** — Accept: `initialize` sends the init handshake; a fed `publishDiagnostics` notification surfaces via `onDiagnostics` as normalized `Diagnostic[]`; `hover` resolves with the server's reply. No real server in tests.
- [ ] **4 · Manager** — Accept: `forFile("a.ts")` spawns the TS server once and reuses it; `forFile("b.py")` uses a different client; `shutdownAll` closes them. (Integration; skip if server binary absent, log the skip.)
- [ ] **5 · Linters + diagnostics** — Accept: a fixture eslint/ruff JSON parses to `Diagnostic[]`; `mergeDiagnostics` dedups; `formatDiagnostics` returns `""` when clean and a `<pi-lens>` block otherwise.
- [ ] **6 · Verify** — Accept: `parseVerify` extracts pass/fail + failing names from a sample `bun test`/`pytest` output across exit codes; `formatVerify` wraps failures in a `<pi-lens>` block. (Module only — wired into the hook in step 8, no tool.)
- [ ] **7 · `lens` tool** — Accept: `lens` with `action: "hover"|"references"|"definition"|"rename"` dispatches to the manager and returns readable text; `rename` requires `new_name`; an out-of-range position or absent server surfaces a clear tool error.
- [ ] **8 · Hooks + config** — Accept: `tool_result` on read/write/edit injects diagnostics + emits `lens:clean|issues`; `agent_settled` runs verify (after `lens:clean`) + emits `verify:*`; `block` mode gates edits on unresolved errors (always allowing edits *to the erroring file*); `/pi-lens` configures servers/linters/verify-cmd/mode.

## Test Strategy

Pure units carry the weight: `framing`, `diagnostics`, linter `parse` functions (with captured real-tool output fixtures), `parseVerify`. The `client` is tested against fake stdio (fed byte strings). The `manager` and real servers are integration-tested when binaries exist, logged-skipped otherwise. Hooks proven in manual smoke.

## Risks / Notes

- LSP is stateful and async — `didOpen`/`didChange` sync must precede requests, and diagnostics arrive asynchronously; the manager must debounce and associate them to the right file version.
- Server availability varies by machine — degrade gracefully (linters-only, or nothing) when a server binary is missing; never hard-fail a read/edit because the LSP server is absent.
- `block` mode can trap the agent (can't edit because errors, can't fix without editing) — the gate must always allow edits *to the erroring file*. Call this out in step 8.
- Absorbing verify keeps plumbing shared, but keep `verify.ts` cleanly separable in case it ever splits back out.
